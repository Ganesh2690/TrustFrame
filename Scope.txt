Executive Summary TrustFrame is an integrity-focused evidence capture and verification platform designed to provide citizens, media organizations, and enterprise institutions with a trusted method to document real-world events. The platform combines cryptographic asset signing, metadata preservation, and a tamper-evident provenance chain to produce shareable evidence reports that establish credibility without requiring specialized apps or technical expertise. This Scope of Work defines the requirements, timeline, and deliverables for a Proof of Concept (POC) build targeting a 14-day development window. The POC is designed to validate one core hypothesis: Do enterprise organizations actually want a trusted way to capture and prove reality? The POC will deliver a functional mobile web application that allows users to upload media (photo/video), automatically capture metadata (timestamp, GPS, device info), generate a SHA-256 cryptographic hash for immutability, tag content to a specific event, and produce a shareable evidence report. This is not a full product build — it is the smallest possible version of TrustFrame capable of proving value in the market. Core Proof Moment: A user records content → the system timestamps and hashes it → generates a shareable evidence report. If users find value in this flow, we have validation. 2. Project Overview & Problem Statement 2.1 The Problem People increasingly need a trusted way to capture and verify real-world events — from vehicle accidents and property disputes to protests, concerts, and personal safety incidents. Existing solutions are fragmented: law enforcement tools like Axon require officer initiation, human rights apps like eyeWitness serve narrow use cases, and enterprise SDKs like Truepic are invisible to end users. None of them offer a zero-friction, citizen-first, event-based evidence capture experience with instant shareable reports. 2.2 The Solution TrustFrame fills this gap with a mobile web application (no app download required) that lets anyone capture, verify, and share authenticated evidence in seconds. The system provides: cryptographic signing at the point of capture, immutable metadata preservation, event-based architecture where multiple contributors can document the same incident, and auto-generated evidence reports that are credible, shareable, and tamper-evident. 2.3 The Vision (Full Product) The full TrustFrame platform will encompass forensic AI analysis (tamper/deepfake detection), a trust-tier scoring system (Verified | Unverified | Manipulated | Synthetic | Inconclusive), identity and liveness verification, blockchain-anchored provenance, C2PA/CAI standard compliance, and enterprise integrations. The POC intentionally defers these capabilities to validate the core user flow first. 2.4 Three Workflow Channels The full platform will serve three distinct workflow channels: • B2B Enterprise: System-agnostic integration with existing security infrastructure — body cameras, surveillance, fleet management. • B2C Events: Consumer-facing evidence capture activated by individual users at events, incidents, or personal safety moments. • Internal Monitoring & Dashboards: Backend management console for the TrustFrame operations team. The POC focuses on the B2C Events workflow as the primary validation target. 4 3. POC Scope & Objectives 3.1 Primary Use Case Event-based evidence capture — protests, concerts, accidents, incidents, property disputes. A user encounters an event, accesses TrustFrame via mobile web (optionally via QR code), uploads photo/video content, and receives a verifiable evidence report they can share. 3.2 Secondary Use Case (Optional) Personal safety / Guardian Mode — a user activates TrustFrame for ongoing personal documentation during a situation where they feel unsafe. This is scoped as a stretch goal only. 3.3 Validation Objectives The POC will be evaluated against the following signals: • Number of media uploads completed through the platform • Number of evidence reports generated and shared externally • Repeat usage — do users return without being prompted? • Inbound interest from partners, investors, or prospective users • Strongest signal: someone asking to use this for their event, or asking what it costs 4. Functional Requirements 4.1 Media Upload & Ingestion • Mobile web-based upload interface (responsive, mobile-first design) • Support for photo (JPEG, PNG, HEIC) and video (MP4, MOV) file types • Maximum upload size: 500MB per file (configurable) • Upload progress indicator with error handling and retry logic • Optional QR code entry point or voice activation — user scans QR or uses voice command to open the upload interface in-browser with no app download or account creation required 4.2 Cryptographic Hashing & Signing • SHA-256 hash generation on every uploaded asset at the point of ingestion • Hash must be computed server-side on the raw binary file, not on compressed/transcoded versions • Hash stored alongside the asset as an immutable fingerprint • Visible hash displayed on the evidence report and in asset metadata • Optional: client-side hash pre-computation for integrity verification during upload (stretch goal) 4.3 Metadata Capture The system must automatically capture and preserve the following metadata at the point of upload: • Timestamp: ISO 8601 format, UTC, captured server-side at ingestion (with client-reported time as secondary) • GPS / Geolocation: Latitude and longitude from device (with user permission), plus reverse-geocoded address • Device Information: Device model, OS version, browser user-agent string • Session Integrity Hash: A composite hash of the upload session (user ID + timestamp + device + asset hash) • File Metadata: Original filename, file size, MIME type, resolution/duration (for video) 4.4 Event Tagging & Grouping 5 • Users can create or join an Event Channel — a shared container where multiple people contribute media to the same incident • Events have: name, description, location, date/time, unique event ID • Media uploaded within an event is automatically associated and time-ordered • Event channels are accessible via shareable link or QR code 4.5 Evidence Report Generation • Auto-generated evidence report for each upload or event, delivered as a web page (shareable link) and optionally as a PDF download • Report must include: media thumbnail(s), SHA-256 hash, timestamp, GPS coordinates with map pin, device information, event context, and chain-of-custody summary • Report must display clear tamper-resistance messaging (e.g., "This media was hashed at the point of capture. Any modification to the original file will produce a different hash.") • Unique report URL with signed, read-only access (no login required to view) 4.6 Share Functionality • One-tap share of the evidence report via native device share sheet (SMS, email, social, messaging apps) • Copyable link to the report • Reports are publicly viewable via link but not publicly discoverable (security through obscurity + signed URLs) 4.7 UX Design Principles The user experience must be radically simple. The entire capture-to-report flow should require no more than 1–2 taps after initial entry. There is no login wall and no account creation. The interface should feel as effortless as opening a phone camera — any friction at this stage kills the validation signal. • Landing screen presents one clear primary action: Capture or Upload • On first use, the user is prompted for a name or phone number as a lightweight identifier — not a login, just a minimal input so submissions can be attributed and the user can later retrieve their reports. This is stored as a session cookie so they are not prompted again on return visits. • Every screen should have a single dominant action — no multi-step wizards, no settings pages, no sidebars • The flow must work for someone who has never heard of TrustFrame and is using it for the first time at a live event under stress 5. Technical Requirements & Architecture (POC) The architecture below is scoped specifically for the 14-day POC build. It is designed to be modular and extensible so that Phase 2+ capabilities (sync engine, AI analysis, blockchain anchoring) can plug in without requiring a rebuild. A separate, more detailed architecture document will be developed for later phases once the POC validates market demand. 5.1 Application Architecture (POC) Component POC Requirement Application Type Mobile web application (Progressive Web App recommended) — NOT a native app Frontend Responsive mobile-first UI; React, Next.js, or equivalent modern framework. Minimal screens, 1–2 tap flow, no login wall. Backend Cloud-native API layer; Node.js, Python (FastAPI/Django), or equivalent API Design RESTful API with JWT-based authentication; GraphQL optional for future phases 6 Component POC Requirement Database PostgreSQL or equivalent relational DB for structured metadata; object storage for media files. Schema must accommodate multi-contributor event data from day one (multiple submissions per event, contributor session IDs, spatial/temporal metadata per submission) so the sync engine can plug in at Phase 2 without a database rebuild. Cloud Hosting AWS, GCP, or Azure — containerized deployment (Docker); CI/CD pipeline Storage Encrypted object storage (S3/GCS/Azure Blob) for media assets with server-side encryption POC Architecture Note: This architecture is intentionally minimal — it covers only what is needed to validate the core capture → hash → report → share flow. The full-product architecture (AI pipelines, sync engine, blockchain layer, enterprise APIs) will be scoped separately once the POC confirms market demand. 5.2 Security & Cryptography • SHA-256 hashing on all uploaded assets (computed on raw binary) • All data in transit encrypted via TLS 1.2+ • All data at rest encrypted (AES-256 or platform-native encryption) • Role-based access control for admin functions • Signed URLs for report sharing (time-limited or permanent, configurable) • Session integrity hashing to bind upload context to the asset 5.3 Provenance & Custody Ledger (POC Level) For the POC, the provenance chain will be maintained in the application database as an append-only event log. Each asset will have a custody record tracking: upload timestamp, original hash, any access events, and report generation events. Blockchain anchoring is deferred to a future phase but the data model should be designed to accommodate it. Architecture Note: The POC data model must be structured so that provenance records can be anchored to a blockchain or distributed ledger in a future phase without requiring a schema migration. Design the custody ledger as an append-only, immutable event log from day one. 5.4 Performance Requirements • Upload processing (hash + metadata capture): < 10 seconds for files up to 100MB • Evidence report generation: < 5 seconds after upload completes • Report page load time: < 3 seconds on 4G connection • System must support at least 50 concurrent uploads during POC testing 6. Media is automatically hashed and stored, creating a verifiable record of the incident. • Property Dispute: Landlord or tenant documents property condition. Photos are hashed and timestamped, producing a credible before/after record. 7.
